# Bug Report: Orchestrator trace_id Overwrite in PenguiFlow Templates

## Summary

The orchestrator code generated by `penguiflow generate` unconditionally creates a new `trace_id` and overwrites the one passed in via `tool_context`. This causes a trace ID mismatch between the playground (which generates and tracks `trace_id_A`) and the planner (which stores trajectories/events under a different `trace_id_B`). The result is that trajectory and event fetches return 404, and the playground UI permanently shows "No trajectory yet" after every completed turn.

The fix has been applied to the **react** template but has **not been propagated** to the other 6 affected templates.

## Root Cause: Detailed Walkthrough

### The Data Flow

1. **Playground generates `trace_id_A`** ŌĆö `playground.py:1368`:
   ```python
   trace_holder: dict[str, str | None] = {"id": secrets.token_hex(8)}  # e.g. "ceade7eec156f42a"
   ```

2. **Playground passes it as a hint** to the wrapper ŌĆö `playground.py:1466`:
   ```python
   "trace_id_hint": trace_holder["id"],
   ```

3. **`OrchestratorAgentWrapper` adopts it** and puts it in `tool_context` ŌĆö `playground_wrapper.py:458,463`:
   ```python
   trace_id = trace_id_hint or secrets.token_hex(8)   # = "ceade7eec156f42a"
   # ...
   "trace_id": trace_id,                               # passed to orchestrator via tool_context
   ```

4. **Orchestrator IGNORES the incoming `trace_id`** and generates a new one ŌĆö e.g. `enterprise/orchestrator.py.jinja:215`:
   ```python
   trace_id = secrets.token_hex(8)   # e.g. "0620344936d288c1" ŌĆö a DIFFERENT value
   ```

5. **`base_tool_context` overwrites `tool_context`** due to merge order ŌĆö e.g. `enterprise/orchestrator.py.jinja:232-236`:
   ```python
   merged_tool_context = {
       **self._tool_context_defaults,
       **(dict(tool_context or {})),   # has trace_id = "ceade7eec156f42a"
       **base_tool_context,            # OVERWRITES with "0620344936d288c1"
   }
   ```

6. **Planner persists trajectory under `trace_id_B`** ŌĆö `react_runtime.py:886-891`:
   ```python
   ctx = trajectory.tool_context or {}
   trace_id = ctx.get("trace_id")     # "0620344936d288c1"
   await saver(trace_id, session_id, trajectory)
   ```
   Backend receives: `PUT /state-store/trajectories/0620344936d288c1` -> 201 Created

7. **Wrapper returns `trace_id_A`** in `ChatResult` ŌĆö `playground_wrapper.py:528-530`:
   ```python
   return ChatResult(
       answer=normalised_answer,
       trace_id=trace_id,   # "ceade7eec156f42a" ŌĆö the wrapper's original value
   )
   ```

8. **Frontend requests `trace_id_A`** ŌĆö but data is stored under `trace_id_B`:
   ```
   GET /trajectory/ceade7eec156f42a  --> 404 Not Found
   GET /events?trace_id=ceade7eec156f42a  --> 404 Not Found
   ```

### Why It Works in the `react` Template

The `react` template (line 209) honors the incoming value:

```python
trace_id = (tool_context or {}).get("trace_id") or secrets.token_hex(8)
```

This extracts the playground's `trace_id` from `tool_context` before the merge happens, so the value in `base_tool_context` matches the one the wrapper will return.

## The Fix

In each affected template, replace:

```python
trace_id = secrets.token_hex(8)
```

With the pattern already used in the `react` template:

```python
trace_id = (tool_context or {}).get("trace_id") or secrets.token_hex(8)
```

This must be applied in both `execute()` and `resume()` methods of each affected template.

### Reference: The Fixed Code in `react` Template

File: `penguiflow/templates/new/react/src/__package_name__/orchestrator.py.jinja`

- **`execute()`** (line 209):
  ```python
  trace_id = (tool_context or {}).get("trace_id") or secrets.token_hex(8)
  ```
- **`resume()`** (line 315):
  ```python
  trace_id = (tool_context or {}).get("trace_id") or secrets.token_hex(8)
  ```

## Secondary Issue: `base_tool_context` Merge Order

Even in templates where `trace_id` is fixed (including `react`), the merge order causes `base_tool_context` to overwrite ALL keys from the incoming `tool_context`:

```python
merged_tool_context = {
    **self._tool_context_defaults,
    **(dict(tool_context or {})),
    **base_tool_context,            # <-- applied LAST, wins all conflicts
}
```

This means any caller-provided values for `tenant_id`, `user_id`, `session_id`, `turn_id`, `task_id`, or `is_subagent` will be silently overwritten by the orchestrator's own values. This is present in **all** templates.

For `trace_id` specifically, the react template works around this by pre-resolving the value before the merge. But the merge order is still inverted from what a caller would expect. Consider whether the intended semantics are:
- **Orchestrator values always win** (current behavior) ŌĆö then callers should not pass these keys
- **Caller values win when provided** ŌĆö then the merge order should be reversed or `base_tool_context` should skip keys already present in `tool_context`

This is a design decision rather than a clear bug, but worth reviewing.

## How to Reproduce

1. Generate a project using any of the affected templates:
   ```bash
   penguiflow generate --template enterprise my-agent
   ```
2. Run the playground:
   ```bash
   penguiflow dev
   ```
3. Send any chat message
4. Observe:
   - Backend logs show `PUT /state-store/trajectories/<trace_id_B>` -> 201
   - Playground logs show `GET /trajectory/<trace_id_A>` -> 404
   - UI shows "No trajectory yet" despite successful completion
