# Plan: Fix trace_id Overwrite & Merge Order Bugs in Orchestrator Templates

## Context

Two related bugs in the orchestrator templates generated by `penguiflow generate`:

1. **trace_id overwrite** — The orchestrator unconditionally creates a new `trace_id` via `secrets.token_hex(8)`, ignoring the one passed in via `tool_context` from the playground. This causes a trace ID mismatch (playground tracks `trace_id_A`, planner stores under `trace_id_B`), resulting in 404s and the UI showing "No trajectory yet."

2. **Merge order** — `base_tool_context` is spread last in the `merged_tool_context` dict, silently overwriting ALL caller-provided values (`tenant_id`, `user_id`, `session_id`, `trace_id`, `turn_id`, `task_id`, `is_subagent`). Callers expect their values to take precedence when provided.

**Note — excluded templates:** The `flow` and `controller` templates also contain `trace_id = secrets.token_hex(8)` (flow line 96, controller line 84), but they are **not affected** by either fix. They use a different architecture — their `execute()` signature takes `tenant_id`/`user_id`/`session_id` directly instead of a `tool_context` dict, and they have no `merged_tool_context` or `base_tool_context`. The fixes below only apply to the 7 planner-based templates.

## Fix 1: trace_id Pre-resolution (6 templates)

In each of the 6 affected templates, replace in both `execute()` and `resume()`:
```python
trace_id = secrets.token_hex(8)
```
with:
```python
trace_id = (tool_context or {}).get("trace_id") or secrets.token_hex(8)
```

Already fixed in `react` template (lines 209, 315).

## Fix 2: Merge Order (all 7 templates, including react)

In all 7 templates (react + the 6 above), change the merge order in both `execute()` and `resume()` from:
```python
merged_tool_context = {
    **self._tool_context_defaults,
    **(dict(tool_context or {})),
    **base_tool_context,
}
```
to:
```python
merged_tool_context = {
    **self._tool_context_defaults,
    **base_tool_context,
    **(dict(tool_context or {})),
}
```

This makes caller-provided `tool_context` values win over orchestrator defaults, which is the expected behavior. The orchestrator's own `base_tool_context` still provides sensible defaults for all keys when the caller doesn't specify them.

**Why both fixes are needed:** Fix 2 alone would resolve the `trace_id` overwrite in the final `merged_tool_context` (since `tool_context` spreads last). However, Fix 1 is still required because `trace_id` is resolved *before* the merge and used to build `base_tool_context` — without Fix 1, `base_tool_context` would contain a freshly generated `trace_id` that doesn't match the caller's, causing inconsistencies in trajectory logging and session tracking even if the final merge gets the right value. Together, the two fixes ensure both internal consistency (`base_tool_context` uses the caller's `trace_id`) and correct merge priority (caller values always win).
